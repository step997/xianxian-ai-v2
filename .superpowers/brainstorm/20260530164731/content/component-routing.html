议题 1 + 3：组件树 & 路由设计

v1.0 是三 Tab 架构（display:none 切换），v2.0 用 React Router 实现 URL 路由

方案 A：三路由 + 共享布局（推荐）

URL: /chat · /persona · /me · 登录页 /login
<App>
├── <AuthProvider>          // JWT 状态上下文
├── <ThemeProvider>         // 主题上下文
├── <Routes>
│   ├── /login → <LoginPage>
│   └── / → <AppLayout>    // 共享壳：侧边栏 + 底部导航 + 宠物头像
│       ├── /chat   → <ChatPage>
│       │   ├── <ChatMessages>      // 消息列表
│       │   ├── <ChatInput>         // 输入框 + 表情选择器
│       │   └── <MobilePetArea>     // 移动端宠物区
│       ├── /persona → <PersonaPage>
│       │   └── <PersonaCardList>
│       │       └── <PersonaCard>   // 人格卡片（可复用）
│       └── /me → <MePage>
│           ├── <ProfileCard>       // 用户信息
│           ├── <ThemeSettings>     // 主题切换
│           └── <SecuritySettings>  // 密码/服务器设置
├── <PetAvatar>            // 侧边栏宠物（PC）/ 底部导航宠物（移动端）
├── <Toast>                // 全局消息提示
└── <Modal>                // 全局弹窗

Pros

  • URL 可分享、可收藏、浏览器前进/后退
  • AppLayout 复用侧边栏+底部导航，不重复渲染
  • v1.0 的 dual-nav 同步问题天然解决（路由驱动）

Cons

  • 切换路由时 AppLayout 内的聊天状态需要保持（用 KeepAlive 或状态提升）
  • 相比 v1.0 多了 URL 对应关系，对用户是心智负担（但很小）

方案 B：单路由 + Tab 状态（接近 v1.0）

URL: /app → 内部用 state 切换 Tab
<App>
└── <MainPage>             // state.currentTab: 'chat'|'persona'|'me'
    ├── <Sidebar />        // PC 导航
    ├── <BottomNav />      // 移动端导航
    ├── <TabPanel tab="chat">
    │   └── ... (同方案A的ChatPage内容)
    ├── <TabPanel tab="persona">
    └── <TabPanel tab="me">

Pros

  • v1.0 迁移最直接，Tab 切换不丢失状态
  • 没有 URL 设计的决策负担

Cons

  • 不能分享/书签特定页面（"把这个人格分享给你"做不到）
  • 不符合 React 生态习惯，后续维护者困惑
  • 浏览器前进/后退不支持
A

推荐方案 A：React Router 多路由

/login · /chat · /persona · /me — URL 驱动，AppLayout 共享壳

B

方案 B：单路由 + 内部 Tab 状态

/app 一个路由，内部 state 切换 — 接近 v1.0 但损失 URL 能力
